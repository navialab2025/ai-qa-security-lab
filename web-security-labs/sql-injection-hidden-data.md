セキュリティ検証ログ：SQLインジェクションによる隠しデータの奪取（Hidden Data Retrieval）の検証
📅 検証実施日
2026年5月23日（再検証・記録）

🕵️‍♂️ 検証概要
Webアプリケーションの商品フィルタリング機能（WHERE句）におけるSQLインジェクション脆弱性の存否を検証。
データベースの条件式を強制的に変更し、本来は「非公開（未リリース）」として隠されている商品データをシステム裏から一括で引きずり出し、画面上に表示させることが可能かを確認した。

🛠️ 検証環境・使用ツール
標的環境: PortSwigger Web Security Academy (SQL injection vulnerability in WHERE clause allowing retrieval of hidden data)

使用ツール: Burp Suite Community Edition

ブラウザ: Embedded Chromium Browser

📑 再現手順（検証プロセス）
Burp Suiteの内蔵Chromiumブラウザから、対象のECサイト（商品一覧画面）へアクセス。

カテゴリ記述のフィルタリング（例: Gifts カテゴリ）を選択した際の、Webブラウザとサーバー間の通信（HTTPリクエスト）をBurp Suiteでインターセプト。

リクエストのURLパラメータ、または検索クエリの末尾に対して、データベースの条件判定を常に「真（True）」にするための以下のハッカーコード（ペイロード）を注入。

Plaintext
' OR 1=1--
（実際の通信上では、URLエンコードされた category=Gifts%27+OR+1%3D1-- として送信）

細工したHTTPリクエストをサーバーへ透過（Forward）させ、ブラウザ側の描画レスポンスを確認。

🎯 検証結果（実際の結果）
画面上に、通常表示されている公開商品だけでなく、本来はステータスが「非公開（未リリース）」に設定されて隠されているはずのすべての商品データが一覧として強制表示された。画面上部に「LAB Solved」のメッセージが出力され、データの不正奪取（漏洩リスク）の実証に成功。

🚨 脆弱性が発生している原因（技術的考察）
アプリケーションがユーザーのカテゴリ選択を受け取る際、内部のSQLクエリを以下のように動的に（文字列結合で）組み立てていることが原因である。

SQL
SELECT * FROM products WHERE category = 'Gifts' AND released = 1
ここに注入したハッカーコードが結合されると、SQL文は以下のように書き換わる：

SQL
SELECT * FROM products WHERE category = 'Gifts' OR 1=1--' AND released = 1
この結果、--（コメントアウト）以降の「AND released = 1（公開済みのものだけ）」という制限ルールが完全に消滅。さらに、OR 1=1 という「絶対に正しい（True）条件」が追加されたため、データベースはすべての商品行を「条件一致」と判断して出力してしまった。

🛡️ 対策・品質保証（QA）エンジニアからの提言
本バグを根本的に修正し、機密データの漏洩を防ぐためには以下の品質改善が必要である：

パラメータ化クエリ（プレースホルダー）の適用:
SQLの組み立てに動的文字列結合を使用せず、カテゴリ名を純粋な「値」として安全に処理する。

最小権限の原則の適用:
Webアプリケーションがデータベースにアクセスする際のユーザー権限を厳格に管理し、未公開データへのアクセス権を必要最小限に制限する。
