セキュリティ検証ログ：SQLインジェクションによる認証バイパス（Login Bypass）の検証
📅 検証実施日
2026年5月23日

🕵️‍♂️ 検証概要
Webアプリケーションのログイン認証機能における、SQLインジェクションの脆弱性（WHERE句のバイパス）の存否を検証。
認証ロジックの不備を突き、正規のパスワードを知り得ない状態から、最高管理者（administrator）アカウントへの不正ログイン（認証バイパス）が可能であるかを確認した。

🛠️ 検証環境・使用ツール
標的環境: PortSwigger Web Security Academy (SQL injection vulnerability allowing login bypass)

使用ツール: Burp Suite Community Edition (プロキシによる通信のインターセプトおよび解析)

ブラウザ: Embedded Chromium Browser

📑 再現手順（検証プロセス）
Burp Suiteを起動し、内蔵のChromiumブラウザから対象のログイン画面（/login）へアクセス。

ユーザー名（Username）入力欄に、認証ロジックを無効化するための以下のハッカーコード（ペイロード）を入力。

Plaintext
administrator'--
パスワード（Password）欄には、任意のダミー文字列（例: 12345）を入力し、ログインボタンを押下。

Burp SuiteにてHTTPリクエストをインターセプトし、パラメータが以下のようにバックエンドへ送信されていることを確認。

HTTP
username=administrator%27--&password=12345
通信をサーバーへ透過（Forward）させ、レスポンスを確認。

🎯 検証結果（実際の結果）
パスワードの不一致による拒否エラーが発生せず、サーバーはリクエストを正常に受け入れ、最高管理者（administrator）としてのログインを許可した。画面上部に「LAB Solved」のメッセージが出力され、認証バイパスの成功を確認。

🚨 脆弱性が発生している原因（技術的考察）
バックエンドのデータベース（SQL）において、ユーザーからの入力を安全に処理する「プレースホルダー（エスケープ処理）」が実装されておらず、入力値がそのままSQL文に結合されている。

内部では以下のようなSQL文が組み立てられていたと推測される：

SQL
SELECT * FROM users WHERE username = 'administrator'--' AND password = '12345'
ハッカーコードに含まれる '（シングルクォーテーション）によってデータ型が強制終了され、さらに --（SQLのコメントアウト構文）によって、それ以降のパスワード検証ロジック（AND password = ...）が丸ごと無効化（無視）された。結果として、ユーザー名のみの完全一致で認証を通過するシステムバグ（脆弱性）が実証された。

🛡️ 対策・品質保証（QA）エンジニアからの提言
この脆弱性を現場で防ぎ、システムの品質を保証するためには以下の実装が必須である：

静的プレースホルダー（バインド変数）の徹底:
ユーザーからの入力を「命令文（コード）」としてではなく、完全に「ただの文字列（データ）」としてデータベースに処理させる。

Webアプリケーションファイアウォール（WAF）の導入・バリデーションの強化:
入力欄に ' や -- などの特殊文字が含まれている場合、フロントエンドおよびバックエンドの入り口で不正な入力として検知・遮断するガードレールを設置する。
